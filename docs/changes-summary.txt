PART C — NEW AWS RESOURCE: DynamoDB Table c3l-nli-user-consent
================================================================================

Table Name: c3l-nli-user-consent
Purpose:    Per-admin consent mapping (student → specific admins)
Region:     ap-southeast-2
Profile:    c3l-analytics

PRIMARY KEY:
  PK (S):  STUDENT#<student_sub>    e.g. STUDENT#uuid-s1
  SK (S):  ADMIN#<admin_sub>#RESOURCE#<domain>#<device>
           e.g. ADMIN#uuid-a2#RESOURCE#health#whoop

GSI1 (for admin queries):
  GSI1PK (S): ADMIN#<admin_sub>     e.g. ADMIN#uuid-a2
  GSI1SK (S): STUDENT#<student_sub>#RESOURCE#<domain>#<device>
  Projection: ALL

Example rows:
  Student uuid-s1 shares health/whoop with Admin2 only:
    PK=STUDENT#uuid-s1, SK=ADMIN#uuid-a2#RESOURCE#health#whoop
    Admin7 has NO record → Admin7 can NEVER see uuid-s1's data.

CREATE TABLE COMMAND (run once):
------------------------------------------------------------
aws dynamodb create-table \
  --table-name c3l-nli-user-consent \
  --attribute-definitions \
      AttributeName=PK,AttributeType=S \
      AttributeName=SK,AttributeType=S \
      AttributeName=GSI1PK,AttributeType=S \
      AttributeName=GSI1SK,AttributeType=S \
  --key-schema \
      AttributeName=PK,KeyType=HASH \
      AttributeName=SK,KeyType=RANGE \
  --global-secondary-indexes '[
    {
      "IndexName": "GSI1",
      "KeySchema": [
        {"AttributeName":"GSI1PK","KeyType":"HASH"},
        {"AttributeName":"GSI1SK","KeyType":"RANGE"}
      ],
      "Projection": {"ProjectionType":"ALL"},
      "BillingMode": "PAY_PER_REQUEST"
    }
  ]' \
  --billing-mode PAY_PER_REQUEST \
  --region ap-southeast-2 \
  --profile c3l-analytics

PART F — WHY LAKE FORMATION WAS REMOVED FROM PER-ADMIN CONTROL
================================================================================

Lake Formation row filters work per IAM role, NOT per user/sub.
All admins share the same IAM role (NextLevelAdminRole).
Therefore: Lake Formation CANNOT distinguish Admin2 from Admin7.

The fix:
  ✅ DynamoDB: Store per-admin consent (student_sub → admin_sub mapping)
  ✅ Lambda:   Filter at application layer (build WHERE IN clause from DynamoDB)
  ✅ Athena:   Executes the scoped query — only sees allowed student_subs
  ❌ Lake Formation: Still used for broad governance (not per-admin)
  ❌ DataZone: Metadata governance only — not for runtime row filtering
