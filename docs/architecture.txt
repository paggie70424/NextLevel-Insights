PLATFORM OVERVIEW
-----------------
NextLevel Insights (NLI) is an enterprise-grade, consent-driven, multi-tenant
data platform that ingests student health and academic data, normalises it to
a stable canonical schema, governs access via Lake Formation, and makes data
available for analytics via Athena — all with explicit student consent.

Naming Convention: All resources prefixed with "c3l-nli-"


================================================================================
ROLES (UPDATED 2026-03-04 — Simplified to 2 Active Roles)
================================================================================

NOTE: Backend AWS DataZone only has admin role. Frontend handles
      admin and student (student) UI states only.

ROLE 1 — STUDENT
  - Owns their own data
  - Logs in via Cognito (group: students)
  - Cognito sub (UUID) = canonical identity — NEVER use email as identity key
  - Clicks "Share with specific admins" in UI
  - Selects which admin(s) to share specific data with
  - Can share with Admin2 and Admin4 but NOT Admin1 and Admin7
  - Consent stored in DynamoDB c3l-nli-user-consent per admin sub
  - Sidebar shows: My Data, Data Resources, Data Permissions, AI Chat
  - Does NOT see Admin section in sidebar

ROLE 2 — ADMIN
  - Logs in via Cognito (group: admin)
  - Cognito sub (UUID) = admin_sub — identity key for consent lookup
  - Sees Admin section in sidebar (purple): "Admin Session" > "Data Dashboard"
  - Dashboard only shows students who explicitly consented to THEIR admin_sub
  - Cannot see data from students who shared with other admins
  - Backend: c3l-nli-AdminDashboard Lambda builds Athena query scoped to
    their admin_sub only (via DynamoDB GSI1 lookup)
  - IAM Role: NextLevelAdminRole

  Step Function: c3l-nli-ConsentShareStateMachine
          |
          +--[1] c3l-nli-ValidateJWT (Lambda)
          |       Verifies Cognito token
          |       Extracts: student_id, role
          |
          +--[2] c3l-nli-StoreConsent (Lambda)
          |       Writes to DynamoDB: nextlevel-consent
          |       PK = STUDENT#<id>  SK = RESOURCE#<domain>#<device>
          |
          +--[3] Map State: ForEach resource (parallel, MaxConcurrency=5)
                  |
                  +-- Choice: health#* → Health Branch
                  |           academic#* → Academic Branch
                  |

                  | c3l-nli-IngestHealth (Lambda)

                  |   Starts Glue crawler (auto-creates if missing)

SUPPORTED DEVICES / LMS SOURCES
================================================================================

Health Domain (device_type values):
  whoop           Whoop API v2
  apple_watch     Apple HealthKit server export
  fitbit          Fitbit Web API v1
  oura            Oura Ring API v2
  garmin          Garmin Health API (placeholder)
  samsung_watch   Samsung Health Partner API (placeholder)

Academic Domain (device_type values):
  canvas_lms      Canvas LMS API
  moodle          Moodle Web Services REST
  blackboard      Blackboard Learn REST (placeholder)

Adding a new device = new device_type value only.
No schema change. No new table. No new data product.
Just extend the switch() in IngestHealth or IngestAcademic Lambda.


================================================================================
DESIGN PRINCIPLES
================================================================================

1. Schema-stable    : Canonical 6-field schema never changes
2. Device-extensible: New device = new device_type string value only
3. Multi-tenant     : student_id in every row + Lake Formation row filter
4. Consent-driven   : No data moves without DynamoDB consent record
5. Governance-first : Lake Formation enforces access, not just IAM
6. DataZone catalog : Business-friendly discovery + subscription workflow
